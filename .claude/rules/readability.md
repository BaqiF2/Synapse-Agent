# 可读性规范

## 目标
让 Agent 生成的代码易读、易理解、易维护，降低后续重构成本。

## 规则

### 1) 命名清晰
- 使用完整语义，避免不必要的缩写
- 布尔值使用 is/has/should/can 等前缀
- 函数名体现动作，变量名体现含义

### 2) 结构简单
- 一个函数只做一件事，超过 ~50 行优先拆分
- 复杂逻辑拆成小函数，并命名表达意图
- 文件组织遵循：imports → types → constants → helpers → exports

### 3) 控制流易读
- 优先使用卫语句（early return）减少嵌套
- 避免超过 3 层嵌套
- 复杂条件用中间变量表达语义

### 4) 类型明确
- 复杂对象/参数必须定义 interface/type
- 对外导出的函数/类显式声明返回类型
- 避免 any；确实无法确定时用 unknown 并做类型收敛

### 5) 注释有价值
- 解释“为什么/规则/边界”，而不是“做什么”
- 复杂算法或业务规则要有简短说明
- 与文件头文档保持一致并及时更新

## 检查清单
- [ ] 命名语义清晰，无随意缩写
- [ ] 函数长度与职责可控，逻辑可拆分
- [ ] 嵌套层级合理，使用卫语句
- [ ] 类型声明清晰，无 any 滥用
- [ ] 注释说明原因或规则，与实现一致
